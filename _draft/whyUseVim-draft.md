저는 평소 빔을 사용하는 개발자입니다. 에디터를 사용할 때는 꼭 빔 플러그인을 설치해서 사용합니다.  
이유는 빔을 사용하기 전과 후로 개발, 문서 편집 환경의 질이 매우 달라졌기 때문입니다.  
이 글은 빔 가이드에 대한 글이 아닙니다. 제가 사용하게된 이유, 어떻게 학습했는지, 장단점은 무엇인지 등을 기록하고 공유합니다.  
그리고 이 글을 통해 평소 빔에 관심있으신 분들께 조심스럽게 권해봅니다.


## 빔을 어떻게 사용하기 시작했는가?
현재 회사에서 애플 매직마우스를 사용하고 있습니다. 저한테는 매직마우스가 맞지 않는지 사용하면서 새끼 손가락과 중지에 통증이 자주 발생했습니다. 

통증이 있다보니 오른손이 예민해졌고, 오른손이 마우스로 이동할 때마다 신경이 계속 쓰이기 시작하여 업무 흐름이 끊기는 일이 많았습니다.

그리고 오른손이 예민해지니 오른손이 자주 키보드를 벗어난 다는 것을 인지하게 됐습니다.

이런 불편함을 개선하기 위해 저는 키보드만 사용해서 모든 작업을 할 수 있는 빔을 공부하기 시작했습니다.


## 발생했던 불편함이 개선됐는가?
결론부터 말씀드리면 매우 좋아졌습니다. 키보드만 사용하여 대부분의 코드 작성 및 수정을 할 수 있게 되었고, 심지어 이전보다 편집 속도가 빨라졌습니다.

그리고 자연스럽게 오른손의 이탈 횟수가 줄어들었고, 마우스 사용을 안하다보니 통증 또한 많이 좋아졌습니다.

불편함들을 개선하고 나니 업무 환경이 매우 쾌적해졌고, 일의 능률 또한 좋아졌습니다.


## 어떻게 개선될 수 있었는가?
빔의 어떤 기능들이 이런 불편함을 개선할 수 있게 했는지 생각해보았습니다. 생각보다 가장 기본적인 기능들이 익숙해졌을 때부터 편리해지기 시작했습니다.

그리고 기본적인 기능들은 빔을 오래 사용해온 지금도 사용하는 기능의 80% 이상을 차지합니다.

그럼 빔에서 사용되는 대표적인 기본 기능들이 무엇이며, 왜 편한지 간략하게 정리해보았습니다.


### 커서 이동
주관적으로 커서 이동만 익숙해져도 빔의 장점을 느낄 수 있다고 생각합니다. 이유는 자음, 모음 자판 위에서 커서를 이동하기 위해 손을 방향키로 이동할 필요가 없어지기 때문입니다.

그리고 조합키로 내가 원하는 디테일한 액션 또한 가능하기 때문에 커서의 이동이 매우 유연합니다.

1. j, k, l, ; 키로 커서 이동.
2. w, b 키를 이용한 문자 단위 커서 점프 기능.
3. $, ^ 키를 이용한 문장 맨 앞, 맨 뒤 커서 이동.
4. I, A 키를 이용한 문장 맨 앞, 맨 뒤 빠른 편집.
5. {, } 키를 이용한 단락 이동.


### 스크롤 이동
빔에서 스크롤 이동이 주는 편리함은 마우스, 트랙패드, 일반 단축키와 비교할 수가 없습니다. 원하는 곳 어디든 키보드 자판위에서 해결할 수 있습니다.

개인적으로 코드 디버깅 할 때 유용하게 쓰인다고 생각합니다.

1. control + e, control + y 를 이용한 스크롤 이동.
2. control + d, control + u 를 이용한 반 페이지 이동.
3. control + f, control + b 를 이용한 페이지 단위 이동.
4. gg, G 키를 사용한 스크롤 맨 위, 맨 아래 이동.
5. number + gg 또는 G 를 이용한 커서 라인 넘버로 이동. (ex. 23gg => 23번줄로 이동)
6. m + [marking] 를 이용하여 위치를 마킹하고, ' + [marking] 을 이용하여 위치로 이동. (개인적으로 a-z로 등록합니다.)

### 문서 편집
빔을 사용하면서 문서편집을 하면 어떤 에디터보다도 빠른 편집을 가능하게 합니다. 에디터 단축키보다 빔 단축키가 더 심플한 것 또한 장점이죠.

1. dd 키로 한줄 전체 삭제.
2. x 키로 단어 삭제.
3. y 키로 복사.
4. p 키로 붙여넣기.
5. u 키로 수정 취소.
6. r 키로 수정 취소 액션 취소.
7. v, V 키로 visual 모드 활성화.


### 키조합
키조합은 빔의 꽃입니다. 앞에 나열한 대표적으로 많이 쓰이는 단축키들만 가지고도 많은 액션을 만들 수 있습니다. 키 조합을 사용하기 시작할 때부터 빔을 이용한 문서 편집 속도가 급격히 상승합니다.

기존에 일반 에디터를 사용하면서 '이렇게 할 수 있으면 좋을텐데'라고 생각했던 기능들을 빔에서는 간단한 키 조합으로 사용할 수 있습니다.

조합키는 정말 응용하기 나름입니다. 위에 단축키가 익숙해지셨다면 숫자 + 단축키를 입력하기 시작하면서 편리한 기능을 찾아가며 써보면 좋을 것 같습니다.


## 어떻게 연습하면 좋은가?
저는 처음부터 순수 VIM만을 사용하지는 않았습니다. 에디터만 쓰던 사용자가 터미널에서 VIM을 열었을 때 당황스러움은 누구나 공감할 것 입니다.

저 또한 느꼈기 때문에 제가 극복해 나갔던 연습방법을 공유해봅니다.


### Editor + Vim plugin
텍스트 에디터 또는 IDE를 사용하다가 빔을 시작하면 자동완성, import 자동화, 글로벌 편집, 디렉토리 트리 등 지원하는 기능이 없으니 막막하고 배울 의지도 꺾입니다.

순수 VIM에서 에디터와 같이 사용려면 .vimrc를 수정해야합니다. 설정할 것도 많고 플로그인도 이것저것 알아봐야합니다.

초심자는 일단 인터페이스가 익숙해지는게 우선이기 때문에 익숙한 에디터에 vim plugin을 적용해서 사용하는 것을 추천합니다. 제가 사용해본 VSCode, JetBrain 계열 IDE, Eclipse, Atom 등은 모두 플러그인이 존재했으니 대부분 있을 것입니다.

하지만 플러그인은 일부 단축키, 명령어 지원 등에서 차이점도 있고 일부 버그도 있는 것을 알고 있으셔야 합니다. 그래도 VIM 인터페이스에 익숙해지는데 문제가 없을 뿐더러 오히려 기존의 에디터 기능을 쓸 수 있으니 순수 빔보다 더 효율적으로 사용할 수도 있습니다.


### vimtutor
```bash
vimtutor
```
터미널에서 위 명령어를 입력하면 빔 튜토리얼이 시작됩니다. 튜토리얼은 빔 에디터로 열리며 가이드를 따라하면서 읽어내려가면 됩니다.

처음에 감을 익히는데 좋습니다. 하지만 재미없으니 2~3번 따라해보고 간단한 코드 또는 문서를 천천히 수정해보는 것을 추천합니다.

위 단축키들이 익숙해지고서 필요한 기능은 인터넷 검색으로도 충분히 익히고 사용할 수 있게 되었습니다.



## 나는 어떻게 사용하고 있는가?
회사에서는 JetBrain 계열의 IDE에 vim plugin을 설치해서 사용하고 있고, 집에서는 터미널만 띄우고 vim + tmux를 사용하고 있습니다.

![my-vim](/assets/images/post/myvim-img.png)*\<Vim + tmux\>*

tmux를 같이 사용하면 vim 명령어로 터미널 명령어를 실행하지 않아도 되고, 서버 구동시 모니터링을 한눈에 할 수 있는 장점이 있습니다.

.vimrc 설정은 처음 사용할 때는 직접 공부하면서 셋팅했지만, 시간도 많이 들고 귀찮아서 [fisa-vim-config](http://fisadev.github.io/fisa-vim-config/)를 수정해서 사용했습니다.

하지만 fisa-vim-config는 많은 플러그인을 적용하고 있고, 제게는 불필요한 설정들이 많아서 대부분 지우고 점점 심플한 빔으로 돌아가고 있습니다.

추가로 키보드는 해피해킹 Type-S 모델을 사용하고 있습니다. 키보드 자체만으로도 좋지만, 빔과 함께 사용할 때 제일 좋은 것 같습니다. 키보드 좋아하시는 분들께 조심스럽게 추천해봅니다.

![happyhacking](/assets/images/post/happyhacking.jpg)*\<해피해킹 키보드\>*


## 장점 말고 단점은?
### 학습 비용이 높다
에디터일 뿐인데 비교적 학습비용이 높습니다. 그래서 시간 투자가 많이 필요합니다. 사실 이 학습 비용 때문에 많은 분들이 중도 포기하는 것으로 알고있습니다.

어떤 사람은 '에디터를 익히는 시간에 실직적인 프로그래밍 기술을 연구하는 것이 더 효율적이다'라고 합니다. 저도 그렇게 생각하기 때문에 단점이라고 생각합니다.(잉?)

### 다른 에디터가 불편해진다
일반 에디터가 불편하게 느껴집니다. 일화로 노션 문서 편집을 할 때 무의식 중에 빔 단축키를 누르게 되더군요.

일반 문서를 작성할 때도 종종 '아 ~ 빔 플러그인 없나?`하고 생각하게 됩니다.

보편화 되어 있는 도구가 불편하게 느껴지는 것은 의외로 큰 단점일 수 있습니다. 대부분의 사람들은 보편화된 도구를 사용하기 때문에 대부분의 사람들에게 맞춰야할 순간이 자주 찾아오기 때문입니다.

### 협업
협업할 때 빔을 사용할 줄 아는 개발자와 함께한다면 문제가 되지 않습니다. 하지만 빔이 익숙하지 않은 개발자가 제 환경에서 타이핑을 할 경우가 생기면 굉장히 불편한 일이 발생할 수 있습니다.


## 마무리하며
처음에는 빔이 좋다고 이야기하다가 마무리를 단점으로 끝내게 되었네요. 하지만 이 단점을 감안하고 사용할 만한 가치가 있다고 말하고 싶습니다.
